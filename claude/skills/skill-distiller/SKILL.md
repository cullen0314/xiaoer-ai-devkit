# Skill Distiller

## 目标

将用户的一次真实经验、重复工作流、提示词模式、判断方法或协作流程，蒸馏成个人可复用的 `SKILL.md` 草稿。

本 Skill 的核心任务不是泛泛总结经验，而是按固定标准提取：

- 输入
- 输出
- 约束
- 执行流程
- 阶段门禁
- 判断规则
- 禁止行为
- 质量检查清单

## 输入

### 必填输入

- 蒸馏目标：用户想沉淀成什么能力。
- 适用场景：未来用户在什么情况下会触发这个 Skill。
- 真实案例：至少一个真实任务案例；如果没有真实案例，只能生成“探索版草稿”。

### 可选输入

- 原始提示词
- 执行过程
- 最终产出样例
- 常见错误
- 输出偏好
- 禁止行为
- 成功标准
- 已有工具、脚本或模板

## 输出

只允许两类输出。

### 信息不足时

输出：

```text
## 上下文扫描

- 目标 Skill：
- 已有材料：
- 可直接使用的信息：
- 缺失信息：

## 缺口判断

- 当前是否能继续：
- 阻塞原因：
- 下一步需要补充：

## 引导问题

1.
2.
3.
```

每轮最多问 3 个问题。

### 信息足够时

输出：

````text
## 蒸馏报告

### 1. 输入摘要
### 2. 可复用流程
### 3. 判断规则
### 4. 输出模板
### 5. 风险与失败补丁

## SKILL.md 草稿

```markdown
...
```
````

不要输出未结构化的散文式总结。

## 约束

- 必须先做上下文扫描。
- 必须先判断输入是否足够。
- 信息不足时，只输出缺口分析和引导问题。
- 没有真实案例时，只能生成探索版草稿，不能生成正式版草稿。
- 每轮最多问 3 个问题。
- 不要询问已经能从上下文推断的信息。
- 不要默认创建文件、目录或脚本。
- 不要把一次性细节固化成通用规则。
- 不要只输出流程，必须包含判断规则。
- 最终草稿必须包含禁止行为和质量检查清单。
- 用户要求“先讨论”“看看效果”“跑一遍样例”时，只模拟执行，不落文件。

## 执行流程

严格按以下 8 步执行。

```text
0. 上下文扫描
1. 输入完整性判断
2. 案例事实记录
3. 可复用步骤抽取
4. 判断规则抽取
5. 输出模板设计
6. 失败补丁设计
7. 生成 SKILL.md 草稿
```

不得跳过第 0 步和第 1 步。

## 阶段门禁

- G1：没有蒸馏目标 -> 必须追问。
- G2：没有适用场景 -> 必须追问。
- G3：没有真实案例 -> 只能生成探索版草稿。
- G4：没有判断规则 -> 不能生成最终草稿。
- G5：没有禁止行为 -> 不能生成最终草稿。
- G6：没有质量检查清单 -> 不能生成最终草稿。
- G7：用户未明确要求创建文件 -> 只能输出草稿。

门禁不通过时，不要继续生成正式版 Skill。

## 阶段产物

### 0. 上下文扫描

输出：

```text
- 目标 Skill：
- 已有材料：
- 可直接使用的信息：
- 缺失信息：
```

### 1. 输入完整性判断

输出：

```text
- 蒸馏目标是否明确：
- 适用场景是否明确：
- 是否有真实案例：
- 当前可生成类型：正式版 / 探索版 / 不能继续
- 下一步动作：
```

### 2. 案例事实记录

输出：

```text
- 任务目标：
- 输入材料：
- 执行过程：
- 最终产出：
- 可复用点：
- 一次性细节：
```

先记录事实，再抽象流程。

### 3. 可复用步骤抽取

每一步输出：

```text
- 步骤：
- 目的：
- 输入：
- 动作：
- 输出：
- 是否必需：
```

只保留可重复使用的动作。

### 4. 判断规则抽取

输出：

```text
- if/then 判断规则：
- 高优先级信号：
- 风险信号：
- 常见误判：
- 需要用户确认的边界：
```

判断规则必须具体到可执行决策，不要只写原则。

### 5. 输出模板设计

输出：

```text
- 标准输出结构：
- 字段说明：
- 空模板：
- 示例输出：
```

模板要稳定、短、可复用，结论靠前。

### 6. 失败补丁设计

输出：

```text
- 易错点：
- 修正规则：
- 必须追问的情况：
- 禁止行为：
- 质量检查清单：
```

### 7. 生成 SKILL.md 草稿

输出完整 `SKILL.md` 草稿。草稿必须包含 frontmatter，并且 frontmatter 只能包含：

```yaml
name: ...
description: ...
```

## 引导提问规则

- 每轮最多问 3 个问题。
- 优先询问会阻塞阶段门禁的问题。
- 提问前先说明当前已知信息和缺口。
- 不要问上下文里已经有答案的问题。
- 不要一次性抛出长问卷。
- 如果用户只给抽象方向，优先追问真实案例和适用场景。
- 如果用户没有真实案例，明确说明只能生成探索版草稿。

## 最终输出格式

信息足够时，最终输出必须使用以下格式：

````text
## 蒸馏报告

### 1. 输入摘要

### 2. 可复用流程

### 3. 判断规则

### 4. 输出模板

### 5. 风险与失败补丁

## SKILL.md 草稿

```markdown
---
name: skill-name
description: ...
---

# Skill Name

## 目标

## 输入

## 输出

## 约束

## 执行流程

## 阶段门禁

## 判断规则

## 输出格式

## 禁止行为

## 质量检查清单

## 触发示例
```
````

如果是探索版草稿，必须在蒸馏报告和草稿标题处标记“探索版”，并列出待真实案例验证的假设。

## 禁止行为

- 禁止跳过上下文扫描。
- 禁止跳过输入完整性判断。
- 禁止在信息不足时生成正式版 Skill。
- 禁止没有真实案例却伪装成已验证流程。
- 禁止只给抽象建议，不给输入、输出、约束和步骤。
- 禁止只给步骤，不给判断规则。
- 禁止把一次性业务细节写成通用规则。
- 禁止默认创建文件、目录、脚本或额外文档。
- 禁止生成 README、安装指南、变更日志等无关辅助文档。
- 禁止为了完整而添加 `scripts/`、`references/`、`assets/`；只有用户明确要求或确有复用价值时才建议。

## 质量检查清单

生成最终草稿前逐项检查：

- 是否包含 frontmatter。
- frontmatter 是否只包含 `name` 和 `description`。
- 是否说明目标。
- 是否定义输入。
- 是否定义输出。
- 是否定义约束。
- 是否有固定执行流程。
- 是否有阶段门禁。
- 是否有判断规则。
- 是否有输出格式。
- 是否有禁止行为。
- 是否有质量检查清单。
- 是否有触发示例。
- 是否区分真实案例、抽象规则和一次性细节。
- 是否能让另一个 AI 在没有额外背景时执行。

## 触发示例

- 把这次产品分析流程蒸馏成 Skill。
- 把这个 bug 定位流程沉淀一下。
- 我想把这套 SEO 写作流程做成 Skill。
- 帮我从这次任务里提炼一个可复用 Skill。
- 这个提示词流程可以蒸馏成 Skill 吗？
- 先按这个方向跑一遍蒸馏看看效果。
- 把“Web 网站 MVP 应该保留什么、砍掉什么”蒸馏成 Skill。
