# TDD 实现 Skill

## 概述

对任务列表中的每个任务执行测试驱动开发，确保代码质量。

## 执行时机

本技能在 `task-list` skill 完成后被调用，即任务列表文档已生成并获得用户批准后。

也可以通过 `/xe:resume` 命令在新建会话中恢复执行。

## 前置检查

**执行前，必须确认以下文件存在：**

```bash
# 检查任务列表文档是否存在
ls docs/plans/YYYY-MM-DD-{feature-name}.md

# 如果不存在，提示用户：
# "未找到任务列表文档，请先运行 task-list skill 或使用 /xe:resume"
```

**如果存在：**
- 读取任务列表文档
- 定位到当前任务编号
- 开始 TDD 开发

## 输入

- 任务列表文档路径（从 `task-list` 传递）
- 当前任务编号（从 1 开始）

## TDD 循环

```
┌─────────────────────────────────────────────────────────────────┐
│                        RED（写失败测试）                        │
├─────────────────────────────────────────────────────────────────┤
│  • 编写描述预期行为的测试                                   │
│  • 测试必须失败（因代码不存在）                               │
│  • 明确测试的输入和预期输出                                   │
└─────────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────────┐
│                      运行测试，验证失败                             │
├─────────────────────────────────────────────────────────────────┤
│  npm test / pytest / ./gradlew test                             │
│  预期：FAIL                                                     │
└─────────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────────┐
│                      GREEN（写最小代码）                         │
├─────────────────────────────────────────────────────────────────┤
│  • 编写刚好使测试通过的最小代码                              │
│  • 不考虑优化、不考虑边界情况                                   │
│  • 目标是让测试通过，不是完美代码                               │
└─────────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────────┐
│                      运行测试，验证通过                             │
├─────────────────────────────────────────────────────────────────┤
│  npm test / pytest / ./gradlew test                             │
│  预期：PASS                                                     │
└─────────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────────┐
│                      REFACTOR（重构改进）                         │
├─────────────────────────────────────────────────────────────────┤
│  • 消除重复代码                                               │
│  • 改进命名和可读性                                             │
│  • 优化性能（在需要时）                                         │
│  • 测试必须保持绿色                                           │
└─────────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────────┐
│                      验证覆盖率                                   │
├─────────────────────────────────────────────────────────────────┤
│  npm test -- --coverage                                       │
│  要求：80%+ 覆盖率                                             │
└─────────────────────────────────────────────────────────────────┘
```

## 执行流程

### 步骤 1：读取状态并验证

```bash
# 读取状态文件，确认 task-list 已完成
node claude/utils/state-manager.js get "{需求名称}"

# 检查 task-list 阶段是否为 completed
# 如果未完成或不存在，返回错误
```

### 步骤 2：读取任务列表

```bash
# 读取 task-list 生成的任务列表文档
Read(docs/plans/YYYY-MM-DD-{feature-name}.md)
```

### 步骤 3：标记阶段进行中

```bash
# 标记 tdd-implementation 阶段为 in_progress
node claude/utils/state-manager.js update "{需求名称}" "tdd-implementation" "in_progress"
```

### 步骤 4：处理当前任务

定位到当前任务编号，读取任务详情：
- 任务名称
- 文件路径
- 代码实现
- 测试代码
- 验证命令

### 步骤 5：执行 TDD 循环

#### RED：编写失败测试

```bash
# 1. 创建测试文件
touch tests/{ComponentName}Test.java

# 2. 编写测试代码
[测试代码]

# 3. 运行测试验证失败
npm test
# 预期：FAIL
```

#### GREEN：编写最小实现

```bash
# 1. 创建源文件
touch src/{ComponentName}.java

# 2. 编写最小实现
[实现代码]

# 3. 运行测试验证通过
npm test
# 预期：PASS
```

#### REFACTOR：重构改进

```bash
# 1. 改进代码
[重构代码]

# 2. 运行测试确认仍然通过
npm test
# 预期：PASS
```

### 步骤 6：验证覆盖率

```bash
npm test -- --coverage
# 要求：80%+ 分支、函数、行、语句覆盖率
```

### 步骤 7：更新进度

```bash
# 使用 decision 记录进度
node claude/utils/state-manager.js decision "{需求名称}" "TDD 进度：已完成 {X}/{N} 个任务"
```

### 步骤 8：更新 TODO 列表

- 标记当前任务为完成
- 移动到下一个任务

### 步骤 9：继续或完成

- 如果还有任务，处理下一个任务
- 如果所有任务完成，进入代码评审阶段

### 步骤 10：标记阶段完成

所有任务完成后：

```bash
# 标记 tdd-implementation 阶段完成
node claude/utils/state-manager.js update "{需求名称}" "tdd-implementation" "completed"
```

**输出完成摘要：**

```markdown
✅ TDD-Implementation 阶段完成

需求名称：{需求名称}
总任务数：{N}
测试覆盖率：{X}%

[Enter] 继续代码执行 | [r] 重新开发
```

- 如果还有任务，处理下一个任务
- 如果所有任务完成，进入代码评审阶段

## TDD 最佳实践

### DO（应该做）

✅ 先写测试，后写代码
✅ 每个测试关注一个行为
✅ 使用有意义的断言
✅ 测试正常路径和边界情况
✅ 模拟外部依赖
✅ 保持测试独立

### DON'T（不应该做）

❌ 先写代码，后补测试
❌ 跳过失败的测试验证步骤
❌ 编写过多代码
❌ 测试实现细节而不是行为
❌ 测试之间有依赖关系

## 测试类型要求

| 类型 | 说明 | 要求 |
|------|------|------|
| **单元测试** | 测试单个函数/类 | 所有公共方法必须有 |
| **集成测试** | 测试 API 端点、数据库操作 | 所有 API 必须有 |
| **E2E 测试** | 测试关键用户流程 | 核心路径必须有 |

## 质量检查清单

- [ ] 测试失败后再实现（RED 步骤未跳过）
- [ ] 只编写足够的代码使测试通过（GREEN 步骤不过度设计）
- [ ] 重构时测试保持绿色
- [ ] 覆盖率达到 80%+
- [ ] 测试独立运行
- [ ] 断言具体且有意义
