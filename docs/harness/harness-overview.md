# Harness 说明

## 1. 什么是 Harness

在 AI Agent 工程语境下，Harness 不是某一个单独的工具、模型或脚本，而是一套围绕 Agent 运行过程构建的**约束、上下文、状态、验证与反馈机制**。

如果把 Agent 理解成“执行者”，那么 Harness 就是它工作的“跑道”和“护栏”。

它解决的不是“模型够不够聪明”的问题，而是：

- Agent 如何在明确边界内工作
- Agent 如何获取正确上下文
- Agent 如何被验证
- Agent 如何在中断后恢复
- Agent 如何和其他 Agent、工具、文档、规则协同工作

一句话概括：

**Harness 的本质，是为 Agent 构建一个可控、可验证、可恢复、可持续演进的运行环境。**

---

## 2. 为什么需要 Harness

单次问答型 AI 主要关注输入和输出，但 Agent 场景会额外面对很多工程问题：

- 任务很长，无法在一个上下文窗口里完成
- Agent 容易一次做太多，导致中途失控
- Agent 可能误判任务已完成
- Agent 可能留下半成品代码或不一致文档
- 多个 Agent 串联时容易丢失上下文
- 项目规范如果只写在文档里，执行时容易漂移

这些问题说明，Agent 不是只靠 Prompt 就能稳定工作的。

当任务开始涉及：

- 多阶段执行
- 多文件改动
- 工具调用
- 跨会话恢复
- 测试与验证
- 多 Agent 协作

就必须引入 Harness。

---

## 3. Harness 的核心目标

一个成熟的 Harness，通常要实现以下目标：

### 3.1 约束执行边界

Agent 不应该在无限自由的环境中随意行动，而应该知道：

- 什么可以做
- 什么不能做
- 什么情况下必须停下来等待确认
- 什么结果才算真正完成

### 3.2 提供结构化上下文

不是把所有信息一次性塞给 Agent，而是给它提供：

- 项目地图
- 文档入口
- 任务状态
- 可调用工具
- 阶段性工件

### 3.3 让执行过程可验证

Harness 要确保 Agent 的“完成”不是口头声明，而是基于：

- 编译结果
- 测试结果
- 引用校验
- 结构校验
- 运行级 smoke test

### 3.4 支持中断恢复

复杂任务很难一次做完，因此 Harness 要提供：

- 状态记录
- 任务进度
- 工件路径
- 失败原因
- 恢复入口

### 3.5 支持持续演进

Harness 不只是一次性搭完，而是需要不断吸收：

- 新的失败模式
- 新的验证规则
- 新的工作流经验
- 新的 Agent 协作方式

---

## 4. Harness 的核心组成

一个完整的 Harness，通常由以下几个部分组成。

---

### 4.1 地图（Map）

地图回答的问题是：

> Agent 去哪里找信息？

常见载体包括：

- `AGENTS.md`
- `CLAUDE.md`
- `docs/architecture/`
- `docs/specs/`
- `docs/plans/`
- `docs/runbooks/`

地图的重点不是记录所有细节，而是建立“导航关系”。

好的地图应该帮助 Agent 快速知道：

- 这个项目有什么目录
- 每类文件应该放在哪里
- 常用工作流入口是什么
- 详细规则去哪里看

---

### 4.2 状态（State）

状态回答的问题是：

> 当前任务执行到哪里了？

常见状态内容包括：

- 当前阶段
- 阶段完成情况
- 已产出的工件
- 当前验证结果
- 最近一次更新时间

常见载体包括：

- `.claude/state/*.json`
- `progress.md`
- `开发任务.md`

状态是实现断点恢复和多 Agent 串联的关键。

---

### 4.3 任务拆解（Task Decomposition）

任务拆解回答的问题是：

> 一个复杂需求如何被切成 Agent 可以稳定完成的小单元？

在没有拆解层时，Agent 容易出现：

- 一次接太多任务
- 半途而废
- 修改范围失控
- 难以验证

因此 Harness 通常要求：

- 先做技术设计
- 再生成开发任务文档
- 每个 task 有明确输入、输出、验证方式

这一步是从“方案”走向“执行”的桥梁。

---

### 4.4 验证（Verification）

验证回答的问题是：

> Agent 怎么证明自己真的完成了任务？

验证不能只依赖自然语言，而应该尽量结构化和自动化。

常见验证方式包括：

- 编译检查
- 单元测试
- 结构校验
- frontmatter 校验
- 路径与引用一致性校验
- 运行级 smoke test

Harness 的关键思想之一，就是把“看起来完成了”变成“验证通过了”。

---

### 4.5 约束（Constraints）

约束回答的问题是：

> Agent 在什么边界内运行？

例如：

- 命名规则
- 文件目录规则
- 前置条件
- 输出格式
- 允许使用的工具
- 必须暂停确认的场景

成熟的 Harness 不会让这些约束只存在于文档中，而会尽量将其机制化、脚本化。

---

### 4.6 反馈回路（Feedback Loop）

反馈回路回答的问题是：

> Agent 犯错之后，系统如何发现、修正并防止重复犯错？

常见做法包括：

- 验证失败后阻断流程
- 审查 Agent 输出
- 沉淀失败模式库
- 持续补充校验规则
- 定期运行清洁 Agent

Harness 的价值不仅在于“防错”，还在于“积累经验”。

---

## 5. Harness 与 Prompt Engineering 的区别

Prompt Engineering 关注的是：

- 如何更好地向模型提问
- 如何组织输入内容
- 如何提升单轮输出质量

Harness 关注的是：

- 如何让 Agent 在真实工程环境中稳定工作
- 如何管理多阶段执行
- 如何管理状态、验证、恢复与协作

两者不是替代关系，而是层级不同：

- Prompt Engineering 更偏局部输入优化
- Harness 更偏系统级运行环境设计

可以理解为：

**Prompt 决定 Agent“怎么想”，Harness 决定 Agent“怎么持续做事”。**

---

## 6. 一个典型的 Harness 工作流

一个典型的 Agent Harness 工作流通常如下：

```text
需求输入
  ↓
读取项目地图
  ↓
生成技术方案
  ↓
拆分任务列表
  ↓
执行具体任务
  ↓
执行验证
  ↓
更新状态文件
  ↓
决定进入下一阶段或暂停
```

在更成熟的场景里，还会额外加入：

- reviewer agent
- clean-up agent
- memory hook
- 文档维护与漂移检查

---

## 7. 好的 Harness 应具备的特征

一个好的 Harness，通常有以下特点：

### 7.1 可导航

新 Agent 进入项目后，能快速知道：

- 去哪找规则
- 去哪找状态
- 去哪找文档
- 从哪开始工作

### 7.2 可拆解

复杂任务能被稳定地拆成小任务，而不是一次性丢给 Agent。

### 7.3 可验证

完成与否有明确验证标准，而不是靠 Agent 自评。

### 7.4 可恢复

中断之后能继续，而不是重新开始。

### 7.5 可扩展

新增 Agent、Skill、Workflow 时，不会把系统搞乱。

### 7.6 可演进

系统会逐渐沉淀出：

- 失败模式
- 更强的约束
- 更好的工作流
- 更清晰的协作边界

---

## 8. Harness 的常见误区

### 8.1 只写 Prompt，不做状态管理

这样一旦任务中断，Agent 就容易失忆。

### 8.2 只写文档规则，不做机械化校验

这样规范会逐步漂移。

### 8.3 一上来就做复杂多 Agent 系统

如果没有状态、验证、工件契约，多 Agent 只会放大混乱。

### 8.4 验证只靠“看起来没问题”

这会导致误判完成，积累信任债务。

### 8.5 地图写成百科全书

把所有信息堆进一个文档，只会增加上下文噪音。

---

## 9. Harness 在本项目中的意义

对于 `xiaoer-ai-devkit` 这类以 Commands、Agents、Skills 为核心的项目，Harness 的意义尤其重要。

因为这个项目本身不是单点功能代码库，而是一个“面向 Agent 的工程能力平台”。

在这样的项目里，Harness 的价值体现在：

- 让不同 Agent 能在统一约束下工作
- 让 command / agent / skill / docs 之间形成稳定关系
- 让工作流可以恢复、验证、扩展
- 让项目不因能力增多而越来越混乱

也就是说，这个项目越往后发展，越不能只依赖 Prompt，而要依赖一整套 Harness。

---

## 10. 总结

Harness 可以理解为 AI Agent 时代的软件工程基础设施。

它的核心不是让模型更聪明，而是让模型：

- 在明确边界内运行
- 在可用上下文中工作
- 在验证机制下交付结果
- 在中断后可以恢复
- 在系统规则中持续演进

一句话总结：

**Harness 不是给 Agent 加功能，而是给 Agent 建环境。没有 Harness，Agent 很难稳定地参与真实工程；有了 Harness，Agent 才可能从“偶尔有用”变成“持续可靠”。**
