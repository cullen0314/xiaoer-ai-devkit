 # AI Agent 常用模式整理

## 概述

AI Agent 的常用模式，本质上是在回答三个问题：

- 怎么思考
- 怎么行动
- 怎么和外部系统或其他 Agent 协作

在实际工程里，这些模式通常不是单独使用，而是组合使用。下面按常见程度和实战价值进行整理。

---

## 1. ReAct 模式

**ReAct = Reason + Act**

核心思想：

- 先思考
- 再行动
- 根据结果继续思考
- 反复循环直到完成任务

### 典型流程

1. Thought：分析当前问题
2. Action：调用工具或执行动作
3. Observation：读取结果
4. Thought：根据结果继续推理
5. 重复上述过程，直到得到最终答案

### 示例

用户提问：

> 帮我查一下某接口为什么 500 了，并给出修复建议。

Agent 可能这样工作：

- Thought：先查看日志
- Action：调用日志查询工具
- Observation：发现空指针异常
- Thought：定位异常对应代码
- Action：搜索代码实现
- Observation：发现某字段可能为空但未做校验
- Final Answer：给出根因分析和修复建议

### 优点

- 适合需要边查边做的问题
- 非常适合带工具的 Agent
- 过程清晰，可解释性强

### 缺点

- 步骤较多，token 消耗较高
- 推理不稳定时，可能在局部循环

### 适用场景

- Debug
- 搜索信息
- 调接口、查数据库、查日志
- 
- 多步任务执行

---

## 2. Plan-and-Execute 模式

核心思想：

- 先整体规划
- 再按步骤执行

### 典型流程

1. Planner 生成计划
2. Executor 按计划逐步执行
3. 必要时重新规划

### 示例

用户提问：

> 帮我给项目加一个导出 Excel 功能。

Agent 可以先形成计划：

- 找到导出入口
- 确认后端数据来源
- 生成 Excel 文件
- 增加下载接口
- 补充测试

然后再逐项执行。

### 优点

- 适合复杂任务
- 有全局视角，结构清晰
- 比纯 ReAct 更容易控制大任务

### 缺点

- 初始计划可能不准确
- 执行环境变化后，计划可能失效

### 适用场景

- 多文件改造
- 新功能开发
- 复杂任务拆解

---

## 3. Reflection / Self-Refine 模式

核心思想：

- 先给出结果
- 再自我审查
- 再进行修正

### 典型流程

1. Draft：先产出初稿
2. Critique：自我评审
3. Refine：基于评审结果优化

### 示例

用户提问：

> 帮我写一个 SQL 优化方案。

Agent 可以：

- 先输出初版方案
- 再检查是否遗漏索引分析、执行计划、数据量评估
- 最后补充完善成较完整版本

### 优点

- 结果质量通常更高
- 适合生成类、分析类任务

### 缺点

- 处理链路更长
- 有时会出现“形式上反思、实际上收益有限”的情况

### 适用场景

- 写代码
- 写文档
- 方案设计
- 代码评审

---

## 4. Tool Use / Function Calling 模式

核心思想：

- 模型不直接凭空回答
- 需要外部信息或执行能力时，调用明确工具

### 常见工具类型

- 搜索
- 数据库查询
- API 调用
- 文件读写
- Shell 命令
- 浏览器操作

### 优点

- 结果更可信
- 能连接真实系统
- 是工程化 Agent 的基础能力

### 缺点

- 高度依赖工具设计质量
- 工具越多，调度复杂度越高

### 适用场景

- 几乎所有生产级 Agent

---

## 5. Multi-Agent 模式

核心思想：

- 一个 Agent 不包打天下
- 将复杂任务拆成多个角色协作完成

### 常见角色划分

- Planner：负责规划
- Researcher：负责检索和调研
- Coder：负责实现
- Reviewer：负责评审
- Tester：负责验证

### 两种常见协作方式

#### 5.1 Manager-Worker

一个主 Agent 负责调度多个子 Agent。

#### 5.2 Peer-to-Peer

多个 Agent 平级协作，彼此传递结果。

### 优点

- 复杂任务更容易拆解
- 分工清晰
- 可以并行处理，提高效率

### 缺点

- 协作成本更高
- 容易出现上下文丢失
- 结果整合难度较大

### 适用场景

- 大型代码库分析
- 复杂研发任务
- 多来源信息整合

---

## 6. Router 模式

核心思想：

- 先判断任务类型
- 再交给最适合的 Agent 或工作流处理

### 示例

用户请求进入系统后，先做分类：

- 问知识 → FAQ Agent
- 查代码 → Code Agent
- 调接口 → API Agent
- 写方案 → Planning Agent

### 优点

- 降低单个 Agent 的复杂度
- 提高处理效率和匹配精度

### 缺点

- 如果路由错了，后续流程也会跟着偏
- 分类规则需要持续优化

### 适用场景

- 平台型产品
- 通用 AI 助手
- 企业内部多场景助手

---

## 7. RAG + Agent 模式

核心思想：

- 先检索相关知识
- 再结合上下文进行推理和执行

RAG 解决的问题是：

> 模型不知道最新信息或私有知识怎么办。

Agent 解决的问题是：

> 拿到知识以后，如何继续做事。

### 示例

用户提问：

> 根据公司内部规范，帮我生成发布流程。

Agent 可以这样处理：

1. 检索内部文档
2. 提取相关规范内容
3. 整理成流程说明
4. 必要时继续调用工具补充信息

### 优点

- 适合接入私有知识库
- 能明显降低幻觉
- 比纯问答更可执行

### 缺点

- 检索质量决定上限
- 上下文拼接方式会影响推理效果

### 适用场景

- 企业知识库助手
- 客服助手
- 内部 SOP 助手

---

## 8. Memory 模式

核心思想：

- Agent 不只依赖当前对话
- 还能利用历史偏好、长期上下文和已知决策

### 常见记忆类型

#### 8.1 短期记忆

当前会话中的上下文。

#### 8.2 长期记忆

跨会话保留的信息，例如：

- 用户偏好中文回复
- 项目技术栈
- 常用命令
- 历史架构决策

### 优点

- 交互连续性更好
- 个性化能力更强
- 减少重复沟通成本

### 缺点

- 可能记错或记住过期信息
- 需要持续治理脏数据和失效记忆

### 适用场景

- 编程助手
- 个人助理
- 长周期任务跟踪

---

## 9. Human-in-the-Loop 模式

核心思想：

- 关键节点由人确认
- Agent 不做完全自动化决策

### 常见需要人工确认的动作

- 删除数据
- 推送代码
- 发布生产环境
- 大范围修改
- 涉及权限、资金或对外可见状态的操作

### 优点

- 更安全
- 可控性强
- 更符合企业治理流程

### 缺点

- 自动化程度降低
- 流程速度会变慢

### 适用场景

- 企业系统
- DevOps 流程
- 安全敏感操作

---

## 10. Event-Driven / Workflow 模式

核心思想：

- Agent 不一定等待用户主动提问
- 也可以由外部事件驱动执行

### 示例

- PR 创建后自动审查
- 告警触发后自动分析日志
- 工单创建后自动分类和分派

### 优点

- 自动化能力强
- 容易接入业务流程

### 缺点

- 依赖外部事件系统
- 错误处理和补偿机制更复杂

### 适用场景

- CI/CD
- 运维告警
- 工单系统
- 企业自动化流程

---

## 模式之间的关系

这些模式可以从三个层面来理解。

### 一、思考模式

决定 Agent 如何思考：

- ReAct
- Plan-and-Execute
- Reflection

### 二、能力模式

决定 Agent 如何获取能力并完成动作：

- Tool Use
- RAG
- Memory

### 三、组织模式

决定 Agent 如何协作、接入系统和控制风险：

- Multi-Agent
- Router
- Human-in-the-Loop
- Event-Driven

---

## 实际项目中的常见组合

### 组合 1：ReAct + Tool Use

这是最常见的基础方案。

适合：

- 查日志
- 查代码
- 调接口
- 现场排障

### 组合 2：Plan-and-Execute + Tool Use + Reflection

适合复杂开发任务。

例如：

- 先规划要改哪些文件
- 再逐项修改
- 最后自检并补充测试

### 组合 3：Router + Multi-Agent

适合平台型助手。

例如：

- 先把请求路由到代码、文档、运维等子 Agent
- 再由对应 Agent 完成任务

### 组合 4：RAG + ReAct

适合企业知识问答加执行场景。

例如：

- 先查询内部知识库
- 再根据知识内容执行后续动作

### 组合 5：Event-Driven + Human-in-the-Loop

适合企业自动化流程。

例如：

- 异常触发后自动分析
- 自动生成修复建议
- 等人工确认后再执行敏感动作

## 工程实践中的选型建议

### 1. 从简单模式开始

不建议一开始就上 Multi-Agent。

推荐优先级：

1. Tool Use
2. ReAct
3. Plan-and-Execute
4. Reflection
5. Multi-Agent / Router

### 2. 一个实用的最小闭环架构

对于多数编程助手或企业助手，一个实用的最小闭环通常是：

- Router（可选）
- ReAct Agent
- Tool Use
- RAG（如果需要接知识库）
- Human confirmation（高风险操作时启用）

### 3. 什么时候不适合上 Multi-Agent

如果只是以下任务，通常单 Agent 就足够：

- 单次问答
- 简单代码修改
- 使用单一工具链即可完成的任务

这类场景中，单 Agent 往往更稳定，也更容易维护。

---

## 一句话总结

如果只记几个最重要的模式，可以记住下面这些：

- **ReAct**：边想边做，适合动态问题
- **Plan-and-Execute**：先规划再执行，适合复杂任务
- **Reflection**：先产出再自查，适合提升质量
- **Tool Use**：让 Agent 真正具备执行能力
- **RAG**：让 Agent 获取私有知识和最新信息
- **Multi-Agent**：通过角色分工处理复杂系统

---

## 结语

AI Agent 并不是某一种固定架构，而是一组可组合的模式。

实际落地时，更重要的不是“用了多少模式”，而是：

- 当前任务是否真的需要这些模式
- 模式之间是否协同顺畅
- 是否能在复杂度、成本和效果之间取得平衡

通常情况下，**能用单 Agent 解决的问题，就不要过早引入多 Agent；能用简单工作流解决的问题，就不要过度设计。**
