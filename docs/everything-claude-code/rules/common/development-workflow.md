# 开发工作流

> 本文件扩展了 [common/git-workflow.md](./git-workflow.md)，包含在 git 操作之前发生的完整功能开发过程。

功能实现工作流描述了开发管道：研究、规划、TDD、代码评审，然后提交到 git。

## 功能实现工作流

0. **研究和重用** _(任何新实现之前的强制步骤)_
   - **首先进行 GitHub 代码搜索**：在编写任何新内容之前运行 `gh search repos` 和 `gh search code` 以查找现有实现、模板和模式。
   - **Exa MCP 用于研究**：在规划阶段使用 `exa-web-search` MCP 进行更广泛的研究、数据摄取和发现先例。
   - **检查包注册表**：在编写工具代码之前搜索 npm、PyPI、crates.io 和其他注册表。优先使用经过实战测试的库而不是手工编写的解决方案。
   - **搜索可适应的实现**：寻找解决 80% 以上问题的开源项目，可以 fork、移植或包装。
   - 当满足要求时，优先采用或移植经过验证的方法，而不是编写全新代码。

1. **首先规划**
   - 使用 **planner** agent 创建实施计划
   - 在编码之前生成规划文档：PRD、架构、系统设计、技术文档、任务列表
   - 识别依赖关系和风险
   - 分解为阶段

2. **TDD 方法**
   - 使用 **tdd-guide** agent
   - 先编写测试（RED）
   - 实现以通过测试（GREEN）
   - 重构（IMPROVE）
   - 验证 80%+ 覆盖率

3. **代码评审**
   - 编写代码后立即使用 **code-reviewer** agent
   - 解决严重和高优先级问题
   - 尽可能修复中优先级问题

4. **提交和推送**
   - 详细的提交消息
   - 遵循 conventional commits 格式
   - 提交消息格式和 PR 流程参见 [git-workflow.md](./git-workflow.md)
