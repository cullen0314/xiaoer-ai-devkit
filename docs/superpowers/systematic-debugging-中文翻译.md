# 系统化调试

## 概述

随机修修补补只会浪费时间，还会制造新的 bug。临时性补丁只是在掩盖底层问题。

**核心原则：** 在尝试修复之前，**永远先找到根因**。只修症状就是失败。

**违背这个流程的字面要求，也是在违背调试本身的精神。**

## 铁律

```
在完成根因调查之前，禁止修复
```

如果你还没有完成第一阶段，就不能提出修复方案。

## 什么时候使用

适用于**任何**技术问题：
- 测试失败
- 生产环境 bug
- 异常行为
- 性能问题
- 构建失败
- 集成问题

**尤其应在以下场景使用：**
- 时间压力很大时（越紧急越容易瞎猜）
- 看起来“只要快速改一下”就行时
- 你已经尝试过多个修复时
- 之前的修复没有效果时
- 你还没有真正理解问题时

**以下情况也不能跳过：**
- 问题看起来很简单（简单 bug 也有根因）
- 你很赶时间（仓促只会导致返工）
- 经理要求立刻修好（系统化处理比来回乱试更快）

## 四个阶段

你**必须**完成当前阶段后，才能进入下一阶段。

### 阶段 1：根因调查

**在尝试任何修复之前：**

1. **认真阅读错误信息**
   - 不要直接跳过错误或警告
   - 它们往往已经包含精确答案
   - 完整阅读堆栈追踪
   - 记下行号、文件路径、错误码

2. **稳定复现**
   - 你能稳定触发问题吗？
   - 具体步骤是什么？
   - 每次都会发生吗？
   - 如果无法复现 → 继续收集数据，不要猜

3. **检查最近的变更**
   - 最近有什么变化可能导致这个问题？
   - 看 Git diff、最近提交
   - 新依赖、配置变更
   - 环境差异

4. **在多组件系统中收集证据**

   **当系统包含多个组件时（如 CI → 构建 → 签名，或 API → 服务 → 数据库）：**

   **在提出修复方案前，先加诊断埋点：**
   ```
   对每一个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置是否正确传递
     - 检查每一层的状态

   先执行一次，收集证据，确认问题具体断在哪一层
   然后根据证据定位失效组件
   再深入调查这个具体组件
   ```

   **示例（多层系统）：**
   ```bash
   # 第 1 层：工作流
   echo "=== 工作流中可用的 secrets: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # 第 2 层：构建脚本
   echo "=== 构建脚本中的环境变量: ==="
   env | grep IDENTITY || echo "环境中没有 IDENTITY"

   # 第 3 层：签名脚本
   echo "=== Keychain 状态: ==="
   security list-keychains
   security find-identity -v

   # 第 4 层：实际签名
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **这样可以揭示：** 到底是哪一层出问题（例如 secrets → workflow ✓，workflow → build ✗）

5. **追踪数据流**

   **当错误出现在调用栈深处时：**

   本目录下的 `root-cause-tracing.md` 提供了完整的逆向追踪技巧。

   **简版要点：**
   - 错误值最初来自哪里？
   - 是谁把错误值传进来的？
   - 持续向上追踪，直到找到源头
   - 在源头修，不要只修症状

### 阶段 2：模式分析

**在修复前先找到模式：**

1. **找到可运行的参考示例**
   - 在同一代码库中找到类似且正常工作的代码
   - 哪些工作正常的逻辑和当前问题最相似？

2. **对照参考实现**
   - 如果你在实现某种模式，必须把参考实现**完整读完**
   - 不要只扫一眼，每一行都要读
   - 在真正理解模式之前，不要套用

3. **识别差异**
   - 正常代码与异常代码的差异是什么？
   - 把所有差异列出来，再小也列
   - 不要主观认为“这点不重要”

4. **理解依赖**
   - 它依赖哪些其他组件？
   - 需要什么设置、配置、环境？
   - 它有哪些默认前提？

### 阶段 3：假设与验证

**按科学方法来：**

1. **只提出一个假设**
   - 明确写出来："我认为 X 是根因，因为 Y"
   - 把假设写下来
   - 要具体，不要含糊

2. **做最小化测试**
   - 只做能验证假设的**最小改动**
   - 一次只改一个变量
   - 不要一次修多个地方

3. **验证后再继续**
   - 成功了吗？成功 → 进入阶段 4
   - 没成功？提出**新的假设**
   - **不要在失败的修复上继续叠补丁**

4. **当你不知道时**
   - 直接说“我不理解 X”
   - 不要假装懂
   - 去求助
   - 继续研究

### 阶段 4：实施修复

**修根因，不修症状：**

1. **先创建失败用例**
   - 用最简单的方式复现问题
   - 能自动化测试就自动化
   - 没有测试框架就写一次性脚本
   - **修复前必须先有这个用例**
   - 可使用 `superpowers:test-driven-development` skill 编写规范的失败测试

2. **实施单一修复**
   - 只针对已识别出的根因
   - 一次只做一个改动
   - 不要“顺手一起优化”
   - 不要捆绑重构

3. **验证修复**
   - 测试现在通过了吗？
   - 有没有破坏其他测试？
   - 问题真的解决了吗？

4. **如果修复无效**
   - **停下**
   - 数一下：你已经试过几次修复？
   - 如果 < 3：回到阶段 1，结合新信息重新分析
   - **如果 ≥ 3：停下，并质疑架构（见下面第 5 步）**
   - 不要在没有架构讨论的前提下尝试第 4 次修复

5. **如果 3 次以上修复都失败：质疑架构**

   **说明可能是架构问题的信号：**
   - 每次修复都会在不同位置暴露出新的共享状态/耦合/问题
   - 修复需要“大规模重构”才能完成
   - 每次修复都会在别处制造新症状

   **这时应暂停，并重新审视基本假设：**
   - 这种模式本身真的合理吗？
   - 我们是不是只是“因为惯性才继续坚持”这个方案？
   - 应该做架构重构，而不是继续修补症状吗？

   **在继续尝试更多修复前，先和你的人工搭档讨论。**

   这不是“假设失败”，而是“架构错了”。

## 红旗信号：立即停下并回到流程

如果你发现自己在想这些：
- “先快速修一下，后面再查”
- “先改改 X 试试看”
- “多改几个地方再跑测试”
- “先不写测试了，我手动验证一下”
- “大概率就是 X，我先修它”
- “虽然没完全搞懂，但这可能能行”
- “模式是这么说，但我想按自己的方式改一下”
- “主要问题有这些：[直接列出修法而没调查]”
- 在没有追踪数据流之前就给出解决方案
- **“再试最后一次修复”**（尤其在已经失败 2 次以上时）
- **每次修复都会在不同位置出现新问题**

**以上全部都意味着：停下，回到阶段 1。**

**如果已经失败 3 次以上：** 开始质疑架构（见阶段 4 的第 5 步）。

## 来自人工搭档的错误信号

**注意这些提醒：**
- “真的会这样吗？”——你在没有验证的前提下做了假设
- “这样能让我们看到……吗？”——你本该先补充证据采集
- “别猜了”——你在还没理解问题时就开始提修复
- “深度想想”——你该质疑基本假设，而不只是处理表面症状
- “我们卡住了吗？”（带有挫败感）——说明当前方法不奏效

**看到这些提醒时：停下，回到阶段 1。**

## 常见自我合理化

| 借口 | 现实 |
|------|------|
| “问题很简单，不需要流程” | 简单问题也有根因。这个流程对简单 bug 也很快。 |
| “这是紧急情况，没时间走流程” | 系统化调试比无头苍蝇式试错更快。 |
| “先试这个，再调查” | 第一个修复动作会定下后续模式，一开始就应做对。 |
| “先确认修好，再补测试” | 没有测试的修复是站不住的，先写测试才能证明。 |
| “一次改多个地方更省时间” | 你无法隔离究竟哪个改动生效，还会引入新 bug。 |
| “参考实现太长了，我按理解改一下” | 只理解一半，几乎必出 bug。必须完整读完。 |
| “我已经看出问题了，直接修吧” | 看到症状 ≠ 理解根因。 |
| “再试最后一次修复” （已经失败 2 次以上） | 3 次以上失败 = 架构问题。应质疑模式，而不是继续修。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|------|----------|----------|
| **1. 根因** | 阅读错误、复现、检查变更、收集证据 | 理解“发生了什么”以及“为什么发生” |
| **2. 模式** | 找可用示例、对比参考实现 | 识别差异 |
| **3. 假设** | 提出理论、最小化验证 | 假设被证实，或得出新的假设 |
| **4. 实施** | 创建测试、修复、验证 | bug 被解决，测试通过 |

## 当流程得出“没有根因”时

如果系统化调查后发现问题确实来自环境、时序或外部系统：

1. 说明你已经完整走完了流程
2. 记录你已经调查过什么
3. 实施合适的处理方式（重试、超时、错误提示）
4. 增加监控/日志，便于后续继续调查

**但是：** 95% 所谓“没有根因”的情况，其实只是调查还不完整。

## 真实世界效果

基于真实调试会话：
- 系统化方法：15～30 分钟修复
- 随机修补方法：2～3 小时来回折腾
- 一次修好的成功率：95% vs 40%
- 引入新 bug 的概率：几乎为零 vs 经常发生
